<Principle>

- 写入纯Markdown内容，不要加代码块标记
- 严格使用图片的实际文件名引用
- 保持与用户输入一致的语言
- **绝不重复引用**：每个参考文献只出现一次
- **严格禁止分点式论述（bullet points / numbered lists）出现在正文中**。必须将分点式内容转换为流畅的论文级段落式语言。
- **避免**口语化表达（"a lot of" → "numerous"）和主观评价词（"very good" → 用数据和“显然”等基于数据的形容支撑）
- 避免过长句子（每句不超过30词）
- 禁止分点列表出现在正文论述中
- 禁止使用`---`分隔线。
- 禁止使用`——`破折号。对于需要破折号加以解释的"大现象——细节"，反转叙事，先写细节，再写大现象。

</Principle>

# 摘要

本节必须包含论文的大标题（`#`）。随便起一个就行。

摘要字样使用`<center><b>摘要</b></center>`。在论文大标题下直接写，不成任何级别标题。

对于所有小问，概述问题，说明方法：罗列模型、结果、分析。说明结论。适当强调加粗其中的着力点、创新点和结论。应当**包含有效而高概括性的真实关键词**而富集信息量。结构：背景(1-2句) → 方法(2-3句) → 结果(2-3句) → 结论(1-2句)。每个问题单独成段，说明用了什么模型、如何构建、得到什么结果（含具体数值）。关键词4~5个。篇幅1页，500字左右。

单独成页面。插入`<div style="page-break-after: always;" />`分页。

**之后的所有正文大章节使用二级标题**

## 问题重述与分析

1. 将题目问题赋予一定背景，适当搜索该题相关背景研究并简要说明当前领域相关研究现状。
2. 将各题目**按已有方案**的视角重述一遍。
3. 分析问题，进行问题的基础假设和类别依据确定。每个问题的分析必须包含模型类型、为什么选择该模型和分析思路概述。

篇幅1~2页。可以适当细说。

## 符号说明

全以表格形式列举。

th：(符号,定义,备注)
td格式：(
$inline-math/symbol$, 
定义应当简短好理解, 
备注如单位等
)

符号包括所有论文中出现的，**不属于熟知的**而是**本论文中定义的符号**典型例子如论文中出现对于数据定义时使用的公式中的字母。定义应当简短好理解。若有复杂说明或量纲写在备注。

## 模型假设

列举题目中任何**未明确说明**的，**本解决方案中假设了的**条件。**不要**在假设中重复题目已有的前提假设条件。

## for 问题X的建模与求解/[step] in steps

对于每个问题，独立开一个大章节二级标题(##)。每个段可选地包含：

### 数据预处理/数据分析/问题分析/根据内容调整

如果该问题有前置处理，或任何建模前的探索性分析，添加此项说明。如一般地，数据型第一问前会进行数据预处理等。

### XX模型的建立

声明模型的结构，列举推导和公式。对于模型的分析要严谨，不绝对化，客观讲述。

论述中应当**尽可能多使用块数学块($$...$$)表达**，同时针对所有数学块给出公式的解释。解释要绝对完整，定义化，学术化，定义最小化。

如果使用同一模型和多种方法/算法，只陈述模型。注意此段**不包含任何方法/算法和结果**，不考虑结果如何，单纯定义数学模型本身。

本块在所有问题步骤中均应该存在。

### 模型求解(与xxx)

在此段中说明求解方法，求解参数与求解结果。

本段中可以涉及具体的求解方法、算法等。但不涉及任何真代码。对于求解方法与算法，可以根据方法数量考虑篇幅地选择性进行简要或详细的数学说明，保证章节内容的充实性。

除非题目要求纯理论且不要求结果，该章节应当存在。

针对简单的结果陈述，结果分析，结果的表/图列举，可以调整标题并一并放入此段。

## 模型的评价与展望

**本节所有标题可以且必须列点陈述**。

### 模型的优点

### 模型的缺点

优点数量多于缺点；缺点约2-3个；每个优缺点都要有具体依据。

### 模型的展望

根据模型解决的具体问题，联想已经解决的问题，考虑扩展到类似问题。基于当前的问题，展望后续的研究。针对不足点，提出新的**无论是否可行但看起来很厉害**的研究方向。这就算展望了。

# 标题规范

论文标题使用一级标题(#);

大标题使用二级标题(##);

所有标题都不加任何序号

正确示例

```
## 问题一的建模与求解
### 基于几何模型的优化模型建立
### 模型的求解与结论
```

错误示例

```
## 五、问题一的建模与求解
### 5.1 基于几何模型……
```

# 图片插入及其规范

**必须在论文中插入代码手生成的图片。** 这是最重要的规则之一。仅仅在文字中写"如图X所示"是不够的，必须同时插入 Markdown 图片标签。

## 插入格式
每张图片必须使用以下格式单独一行插入：
![图片描述](文件相对路径.ext)

一般地，自从当前的`text/paper/[step].md`开始，图片的相对路径以`../../code/[step]/img/`开头。

## 插入规则
1. **可用图片列表中的每张图片都必须在论文中插入**
2. 图片标签必须独占一行，放在相关段落之后
3. 在图片标签之前或之后，必须有至少3行文字对图片内容进行解读分析，或作为一段论述后的证明。
4. 禁止只写"如图所示"而不插入图片标签。
5. 图号引用必须使用如`如::ref{fig:q1_eps_curve.png}`的格式。其中`q1_eps_curve.png`是图片不包含路径的文件名。这将根据事实渲染为`如图 3-1 所示`。
6. 在每个图下方使用 <figure><figcaption>四类区域的出生率与死亡率三十年演化轨迹</figcaption></figure> 写图的标题。无需编号和缩进，有自动样式。

示例：
```
相关性分析结果表明，母亲EPDS得分与婴儿睡眠时长呈显著负相关（r=-0.35, p<0.01），
HADS焦虑得分与睡醒次数呈正相关（r=0.28, p<0.05）。

![相关性热力图](../../code/q1/img/fig1_correlation_heatmap.png)

从 ::ref{fig:q1_correlation_heatmap.png} 可以观察到，母亲心理指标之间也存在较强的共线性关系...
```

**每幅图表至少配3行分析**，说明趋势、原因、与结论的关系，禁止"只放图不解释"。
图表描述必须基于代码手输出的实际数据特征，禁止编造数值。
